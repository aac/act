# Brief Review Round 3

Stability check on brief-v3. Only flagging NEW substantive issues introduced by round-2 incorporations or that round-2 marked Incorporated but where the v3 text fails to pin down behavior.

---

1. **Hook failure semantics directly contradict the "after commit" placement.**
   - **Why it matters:** The Hooks section says hooks "are plain executables run synchronously *after* the corresponding op is committed" and in the same paragraph says "non-zero exit fails the op and rolls back the staged op file *before commit*." These are mutually exclusive: if the hook runs after commit, there is nothing staged to roll back; if it rolls back a staged file, it runs before commit. Round 2 issue #2 explicitly noted the rollback question and the brief now answers it both ways. Two implementers will build incompatible orderings — one will run hooks post-commit and either log-only on failure or do a `git revert`, the other will run hooks pre-commit on the staged op file. A `post-close` Slack hook will either fire reliably and leave broken state on failure, or fire only on success and surprise the user with rollbacks.
   - **Suggested resolution:** Pick one and write it once. Recommended: hooks run after the op file is staged but *before* the op-commit; non-zero exit aborts the commit and unstages the file; rename the phase to `pre-commit-op` in env (`ACT_HOOK_PHASE=pre-commit-op`) so future post-commit hooks (e.g. `post-push`) have a clean slot. Drop the "after the corresponding op is committed" phrasing.

2. **`core.hooksPath=/dev/null` is not portable to Windows.**
   - **Why it matters:** Windows is in the distribution matrix (macOS intel/arm, Linux intel/arm, Windows). `/dev/null` is not a valid path on Windows — git for Windows accepts it in some contexts via msys translation but `core.hooksPath=/dev/null` specifically has been observed to error or to be interpreted as a literal relative path `dev/null` under the repo root, which silently fails to disable hooks (because the directory doesn't exist, git falls through to defaults on some versions and errors on others). This is the bypass mechanism for repo pre-commit hooks added in round 2; if it doesn't work on Windows, every Windows op-commit triggers the host repo's pre-commit pipeline, which is exactly the failure round 2 issue #3 was meant to prevent.
   - **Suggested resolution:** Use `git -c core.hooksPath=` (empty) or `git commit --no-verify` instead. `--no-verify` is the documented, cross-platform way to skip pre-commit/commit-msg hooks; the brief's stated rationale ("op commits touch only `.act/ops/**`") applies identically. Reserve `core.hooksPath` for cases where you also need to skip post-commit hooks (which the brief doesn't currently call out).

3. **Squash-before-push silently invalidates `closed_by_commit` reverse-index entries.**
   - **Why it matters:** Brief-v3 specifies two new behaviors that interact: (a) "Before push, `act` collapses contiguous `act-op:` commits into a single squashed commit," and (b) snapshots carry a `closed_by_commit` reverse index so doctor can verify symmetrically across squash and rebase (round 1 issue #8 fix). But (a) rewrites the very commit SHAs that (b) records. If a `close` op is auto-committed locally as commit `abc123`, the snapshot's `closed_by_commit=abc123` is written into the snapshot op payload; then the squash-before-push rewrites that commit to `def456`, and the recorded SHA now points to nothing on origin. Doctor's symmetric check fails on every fresh clone after the first squash. The two round-1 fixes are individually correct but compose into a footgun.
   - **Suggested resolution:** Either (i) populate `closed_by_commit` lazily — only after squash-and-push, recording the post-squash SHA — and have doctor tolerate empty values for never-pushed snapshots; or (ii) drop SHA-based `closed_by_commit` and record `closed_by_tree` (the git tree hash of `.act/ops/<id>/` at the close, which is invariant under commit squash); or (iii) skip squash for any commit range containing a close-op. Pick one; document.

4. **The importer's id-mapping file has no specified location, format, or lifecycle.**
   - **Why it matters:** Round 2 issue #5 explicitly asked for either deterministic ids or "an id-mapping file the user can grep against early commit messages." Brief-v3 takes the mapping-file path: "emits an id-mapping file." But the brief does not say where it lives (`.act/import-mapping.json`? `_generated/projects/act/mapping.json`? stdout?), what its format is (jsonl? json object? csv?), whether it is committed to git (it must be, if downstream commits reference pre-import ids and a teammate clones later and needs to resolve them), how multiple imports compose (one mapping file per import? appended?), or when it can be garbage-collected. This is the only path into v0.1, so two implementers building from this brief will produce mapping files that are mutually unreadable, and the bootstrap workflow that the brief markets as load-bearing breaks across machines.
   - **Suggested resolution:** Pin: mapping file lives at `.act/imports/<import-timestamp>.json`, committed alongside the imported ops, format `{"source": "issues.jsonl@<sha>", "mapping": {"bootstrap-id-1": "act-a1b2...", ...}}`. Importer is idempotent on re-run (skip already-mapped source ids). `act show` and `act log` accept either pre- or post-import ids and resolve via the mapping files.

5. **The `act version --check-repo` / `writer_version` upgrade gate has no specified behavior for the squash-and-push path.**
   - **Why it matters:** Round 1 issue #15 was incorporated as: ops carry `writer_version`, readers exit "upgrade required" if any op's writer_version exceeds reader. But brief-v3 also specifies that before push, `act` squashes contiguous op-commits. The squashed commit's metadata (author, message) belongs to whichever `act` invocation did the squash — yet the squash spans commits potentially produced by *different binary versions* on the same machine over time (a developer upgrades `act` between sessions but pushes once). The op file payloads each carry their own `writer_version`, fine. But there's no specification of what `writer_version` the squash commit message advertises, nor whether the squash itself is gated by `act version --check-repo` against the union of versions in the commits being squashed. A v0.5 binary squashing v0.7 op-commits silently pushes v0.7 ops to origin under a v0.5-authored squash commit; a downstream reader pulling and running `act version --check-repo` sees the v0.7 ops and correctly errors, but the squash-author signal is misleading for debugging.
   - **Suggested resolution:** Spec: `act` refuses to squash a commit range whose op payloads include `writer_version` newer than the running binary (force-upgrade path); the squashed commit message carries `act-squash: writer_version=<max-of-range>` so downstream readers and `act doctor` can correlate. Alternatively, drop pre-push squash entirely for ranges that span versions. Either is fine; pick one.
