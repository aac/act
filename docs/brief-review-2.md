# Brief Review Round 2

1. **HLC has no defined node-id source, so tiebreaks aren't actually deterministic across machines.**
   - **Why it matters:** A hybrid-logical clock is `(physical, logical, node_id)`. The brief specifies HLC plus op-hash tiebreak but never says what the HLC's node component is or how it's seeded. Two machines that boot identically (e.g. two ephemeral Cowork containers, or CC-on-the-Web sandboxes that share an image) can produce identical `(physical, logical)` pairs for ops created in the same millisecond; if op-hash is the only tiebreaker and the op payloads happen to match (two agents both writing `status=in_progress` with the same HLC), you get identical filenames and one op silently overwrites the other on `git pull`. The "refuse ops with >5min HLC delta" rule presumes a local reference clock, but the brief doesn't say what that reference is — local wall-clock? last-seen HLC from the repo? On a fresh Cowork container with skewed clock, the first op's HLC will fail its own delta check against `now()`.
   - **Suggested resolution:** Specify the node-id explicitly: a per-installation UUID stored in `.act/config.json` (or derived from `$HOSTNAME + $USER + a random init nonce`), included in the op payload and in the filename hash so collision is structurally impossible. Define the HLC plausibility check against `max(local_wall, last_seen_hlc_in_repo)` rather than local wall alone, so a fresh container catches up to repo time on first read instead of refusing every write.

2. **Hooks are "synchronous after commit" with no specified failure semantics.**
   - **Why it matters:** Hooks are now in v1, but the brief gives one sentence: "plain executables run synchronously after the corresponding op is committed." Unspecified: (a) what happens if the hook exits non-zero — does the op get reverted? does `act create` exit non-zero? (b) what's the timeout? a hung `post-claim` hook hangs every claim forever; (c) what env/args does the hook receive — issue id? full op JSON on stdin? working directory? (d) does the hook run during `act import` (replaying 200 issues from the bootstrap JSONL fires 200 post-create hooks)? (e) does it run during fold/replay on a fresh clone? Two implementers will build incompatible hook contracts; a user's `post-close` script that posts to Slack will either fire once, fire on every clone, or fire never depending on which interpretation wins.
   - **Suggested resolution:** Spec the contract: hook receives op JSON on stdin and `ACT_ISSUE_ID`/`ACT_OP_TYPE` env vars; runs only on the writer that produced the op (never during fold/replay/clone); non-zero exit logged to `.act/hooks.log` but does not revert the op (op already committed, can't be unwound); default timeout 30s, killed with SIGTERM then SIGKILL; explicit `--no-hooks` flag on `act import` and any other replay path.

3. **Auto-commit-per-op interacts badly with repo pre-commit hooks.**
   - **Why it matters:** The brief commits to auto-commit-per-op as default ("durability beats history aesthetics") and also lists an opt-in `act` pre-commit hook for state validation. But the host repo almost certainly has its own pre-commit hooks (linters, formatters, secret scanners, test runners) that run on every `git commit`. Every `act create`/`act update` will now invoke the host repo's pre-commit pipeline — potentially seconds per op, potentially failing on unrelated working-tree state ("you have a lint error in `src/foo.ts`, commit aborted") which kills the op-commit and leaves the op file uncommitted but on disk. Squash-before-push doesn't help: by then the op is already lost. Agents will see mysterious "claim succeeded but didn't push" failures.
   - **Suggested resolution:** `act` op-commits must bypass repo hooks (`git commit --no-verify`) by default, with rationale: op commits touch only `.act/ops/**` and are validated by `act doctor`, not by repo lint. Document this clearly. Provide `--verify` opt-in for users who explicitly want repo hooks to run on op commits. Alternatively: stage op files but defer the actual commit to a `git commit -m act-op:...` invoked at end-of-command, and add a "uncommitted ops" check to `act doctor`.

4. **`act_next` composition under contention can leave the agent holding a claim it didn't ask for.**
   - **Why it matters:** `act_next` = ready + claim + show. The claim sub-step is now blocking-and-verifying: on loss it exits non-zero with `{claimed: false, winner: ...}`. But `act_next`'s natural retry behavior is "if I lost issue A, try issue B" — and the brief doesn't say whether the losing claim op stays in the log. It does (append-only). So a flapping `act_next` on a contested ready queue writes a trail of losing claim ops, each of which is real history. Worse: since claim is `assignee=$user + status=in_progress` atomically, the *winner's* state is `in_progress` and the loser's loss op was overridden by HLC — but two agents both trying `act_next` against the same single ready issue will each retry against the next ready issue, and now a chain of issues are getting churned with claim-loss ops. There's no specified backoff, no specified "skip issues I just lost." Two implementers will build different retry/backoff policies and produce wildly different op-log shapes.
   - **Suggested resolution:** Specify `act_next` retry semantics: on claim loss, the composed tool refolds the ready queue (excluding the just-lost issue), retries up to N times (default 3), with jittered backoff; returns structured `{claimed_issue: null, attempts: [...]}` if exhausted. Alternative: make claim attempts via `act_next` not write a loss op at all — claim by writing op only after a speculative re-fold says we'd win, accepting a small race window. Pick one and document.

5. **Importer JSONL schema is unspecified and is the only path into v0.1.**
   - **Why it matters:** The bootstrap is now load-bearing: a 50-line shell script writes `_generated/projects/act/issues.jsonl`, and v0.1 imports it. But there's no schema for the JSONL — is each line a full issue, or an op? If an issue, how are deps and parent represented (by title? by some pre-assigned id?) given that real ids are content hashes the script can't compute the same way the binary will? If an op, the script is now reproducing the op format the binary hasn't shipped yet. If the script assigns its own ids and the importer remaps them, the build issues' references in commit messages and PR descriptions point to ids that won't exist after import. Two implementers will write incompatible bootstrap scripts and incompatible importers.
   - **Suggested resolution:** Pin the JSONL as a sequence of `create` and `dep-add` op payloads in the same op schema v0.1 will use (minus filename — importer assigns HLCs and ids on import). Specify that the script computes ids deterministically using a documented hash recipe so references in the script's own output match post-import ids. Or: importer rewrites references in a stable ordering and emits an id-mapping file the user can grep against early commit messages.

6. **`redact` op's interaction with derived state and re-folds is incoherent as written.**
   - **Why it matters:** "A `redact` op overwrites prior op payload contents in derived snapshots while leaving a tombstone in the op log." But the op log is the source of truth, and folds are reproducible. If a redact op only affects "derived snapshots" (the snapshots/ files), then a fresh clone that folds from raw ops still reproduces the secret — defeating the purpose. If redact actually rewrites the on-disk op file payload, the brief's "append-only, never modified" invariant breaks and op-hash filenames become stale. Combined with reproducibility-as-byte-identical-JSON-output: a redacted field renders one way pre-redact and another way post-redact, so fold output isn't byte-identical across the redact boundary unless redact is applied during fold itself. The brief doesn't say redact is fold-applied.
   - **Suggested resolution:** Specify that redact is fold-applied: during fold, when a redact op is encountered, the named earlier op's payload fields are replaced with `<redacted>` markers in the in-memory state (and therefore in derived snapshots and query output). The on-disk op file is untouched; secret remains in git history; documented escape hatch is `git-filter-repo` (already stated). State the consequence: `act log <id>` after a redact shows the tombstone but the original op file content is still recoverable from git. This is consistent with everything else but needs to be written down because the current text suggests an irreconcilable mix.

7. **`--push-always` and the default-no-push-on-loss leave a class of ops stranded locally.**
   - **Why it matters:** The claim protocol says "push only on win unless `--push-always`." But losing claim ops are still committed locally (per auto-commit default). They sit in local history, never pushed, divergent from origin. Next non-claim op (e.g. the agent goes and works on a different issue, runs `act update --description`) auto-commits and presumably auto-pushes (the brief doesn't actually say when push happens for non-claim ops). If non-claim ops auto-push, they'll push the prior unpushed loss-commits along with them — fine — but if non-claim ops also use the "push on success" gate, you accumulate unpushed local ops indefinitely. The brief doesn't define the default push behavior for non-claim ops at all.
   - **Suggested resolution:** State the push policy explicitly: by default, every auto-commit is followed by `git push` (with rebase-on-reject retry); `--no-push` skips. The claim protocol's "push only on win" is a special case that exists because pushing a losing claim wastes a network round-trip and pollutes origin with no-op history; alternative is to amend-out the loss commit locally before any subsequent push (but that violates append-only). Pick one and document. Also state behavior when there's no configured remote (offline / `--isolated`): auto-commit yes, auto-push no, no error.

8. **`act doctor`'s `index-divergence` check is circular without a defined oracle.**
   - **Why it matters:** `index-divergence` presumably means "SQLite index disagrees with op-log fold." But the SQLite index *is* the fold output cached. To check divergence you re-fold and compare; if they disagree, the cache is stale and the fix is to invalidate. So the check is really "is the fold-checkpoint fresh?" — which is already enforced by tree-hash matching at every read. There's no independent oracle. Either this check is redundant with the tree-hash gate, or it's checking something else (e.g. SQLite schema drift after a binary upgrade) that needs naming.
   - **Suggested resolution:** Replace `index-divergence` with two narrower checks: `index-schema` (SQLite schema version matches binary's expected schema; rebuilds if not) and `index-stale` (fold-checkpoint tree-hash matches current; rebuilds if not). Or drop the check from doctor and make it implicit on every read (which the brief already implies). Picking nothing means two implementers build different checks under the same name.
