# Spec Review Round 1

Consolidated from four parallel reviews of /home/user/act/docs/spec.md sections.

## A. Data model (spec.md L1-300)

1. **Short-id length N starts at 4 but display uses shortest-unique-prefix; which is stamped into op `issue_id`?** Resolution: op envelope `issue_id` is the on-disk stamped id from collision-extension (>=4 hex); display layer recomputes shortest-unique prefix per invocation purely for printing.
2. **`remove_accept` shifts later indices down, but `redact` references `acceptance_criteria[2].text` by index — which index space?** Resolution: redact `field_path` indices refer to the post-fold current index at the redact op's HLC; folding a later `remove_accept` does not retroactively renumber prior redacts (the redact's effect is recomputed against current indices each fold).
3. **`hlc.wall` is RFC3339 string in envelope but `int64_ms_unix_utc` in HLC state — which is canonical for ordering?** Resolution: ordering uses parsed milliseconds-since-epoch from the RFC3339 string; the string form is the wire format and MUST round-trip exactly to the int64 ms value (no sub-ms precision allowed).
4. **`update_field` allows `field: "status"` but `claim` and `close` also mutate status — can a raw `update_field status=closed` bypass close semantics?** Resolution: `update_field` with `field=status` is rejected at write time for values `closed` and `in_progress`; those transitions MUST go through `close` and `claim` ops respectively.
5. **Op file `<hash8>` collision extends to 12/16 hex, but the filename is also content-addressed — does extending the hash require rewriting the file?** Resolution: the filename hash length is chosen before write by probing the target shard directory under `.act/.lock`; the file is written exactly once with the final-length hash in its name, and the hash is purely a filename disambiguator (not stored inside the envelope).

## B. Op-fold (spec.md L250-500)

1. **`op_hash` input ordering is under-specified.** The line `sha256(canonical_json(op.payload || op.hlc || op.node_id))` uses `||` which one implementer may read as logical-or (taking the first truthy field) and another as concatenation/tuple of all three. Resolution: define `op_hash = sha256(canonical_json({payload, hlc, node_id}))` over a fixed-key canonical object.

2. **HLC `receive` is invoked during fold but fold is pure.** Section 1.4 defines `receive(now_ms, msg_hlc, prev)` advancing local HLC, yet section 2's `fold` never calls it and is required to be deterministic across machines. One implementer may mutate `last_hlc` during fold (nondeterministic via `now_ms`); another may skip it. Resolution: fold never calls `receive`; HLC advancement happens only at op-write time, and `last_hlc` is refreshed post-fold from `max(op.hlc)`.

3. **Claim winner tiebreaker conflicts with LWW tiebreaker.** Section 4 picks the *smallest* `(wall, logical, op_hash)` while section 3 resolves all other fields by *greatest* tuple, and `claim` ops also set `(assignee, status=in_progress)`. Two implementers will disagree on whether the assignee field follows claim-earliest or LWW-latest when a later non-claim `assignee` write exists. Resolution: claim winner uses earliest-tuple; the resulting `assignee`/`status` write is materialized as a synthetic LWW write stamped with the winning claim's HLC, and any later LWW `assignee` op overrides it.

4. **`closed` terminal rule vs. LWW for `closed_at`/`closed_reason`.** Section 3 says transitioning out of `closed` is ignored except via `reopen`, but `closed_at`/`closed_reason` are "LWW thereafter, but only writable by ops whose payload also carries `status=closed`". After a `reopen`, it is unspecified whether `closed_at` is cleared, retained, or remains LWW-eligible. Resolution: `reopen` clears `closed_at` and `closed_reason` and resets their `last_hlc` to the reopen op's HLC; subsequent `status=closed` ops may overwrite them again.

5. **Checkpoint per-issue reuse iterates `known_issues + new_issues` ambiguously.** The pseudocode does not define how `new_issues` is discovered when `cp.tree_hash != current` without doing a full directory scan, nor what happens to issues present in `cp` but absent on disk (deleted shard). Implementer A may treat missing subtrees as cache-keep; implementer B may drop them. Resolution: enumerate `new_issues` via `git diff-tree cp.tree_hash current -- .act/ops/`; issues whose subtree is absent in `current` are dropped from the new checkpoint and refolded as empty (tombstone-only state preserved through ops, not cache).

## C. Commands (spec.md L500-750)

1. **`act show <id>` exit code on unknown id conflicts with universal pipeline.** The pre-import id resolution section says zero prefix matches yields exit 3 ("not found"), and `show`'s exit codes list "3 if id not found" — but exit 3 is also defined universally as "environment error" (not in git repo, `.act/` missing). One implementer may return 3 for not-found, another may return 1 (logical failure: "no such issue"). Resolution: not-found is exit 3 per the resolution pipeline; document explicitly that 3 covers both env errors and resolution misses.

2. **`act update --status closed` vs `act close` idempotency divergence.** `update` says `--status closed` is exit 2 (must use `act close`), while `close` on an already-closed issue is idempotent (exit 0, no op). One implementer may treat `update --status closed` on an already-closed issue as exit 2 (usage), another may short-circuit to exit 0 (already-closed idempotent). Resolution: exit 2 unconditionally — `update` never accepts `closed`, regardless of current state.

3. **`--claim` HLC drift check ordering vs pull-rebase.** Edge cases say "HLC drift >5min from repo reference refuses the op (exit 1)", but it's unspecified whether the drift check runs before or after the implicit `git pull --rebase`. One implementer may check local HLC pre-fetch (cheaper), another post-fetch (more accurate reference). Resolution: check HLC drift before the pull-rebase, against the last known repo reference, to fail fast without network mutation.

4. **`act create --parent` to a closed issue: warning channel under `--json`.** Edge cases say closed-parent is "allowed with a stderr warning" but `--json` semantics for warnings are unspecified. One implementer may emit human text on stderr while JSON goes to stdout; another may suppress the warning entirely under `--json` or embed a `warnings: [...]` field. Resolution: under `--json`, embed `"warnings": ["parent_closed"]` in the stdout JSON object and suppress stderr text.

5. **`act dep add` duplicate-edge idempotency vs `--type` mismatch.** Edge cases say "duplicate edge is idempotent (no op, exit 0)", but it's unspecified what counts as duplicate when `--type` differs (e.g., existing `relates` edge, new request `blocks` between same pair). One implementer may treat (child,parent) as the dedup key and exit 0; another may treat (child,parent,type) as the key and write a second op. Resolution: dedup key is (child, parent, type) — different types between the same pair are distinct edges and produce new ops.

## D. MCP / errors / tests (spec.md L750-end)

1. **`act_next` claim-loss backoff jitter sign and distribution unspecified.** The spec says "±25% jitter" on 100ms/400ms/1.6s but doesn't define whether jitter is uniform, gaussian, or signed-symmetric, nor whether each attempt re-rolls independently. Resolution: uniform random in `[0.75x, 1.25x]` of the base delay, re-rolled per attempt.

2. **`act_block` atomicity is asserted but undefined under partial failure.** The composed tool writes `set-status` then `dep-add` and the test asserts "atomically", but there's no rule for what happens if the second op fails (rollback the first? leave it? emit which error?). Resolution: write both ops to a staging area, then `git add` + commit both in a single commit; on any failure, unstage and delete both, surface the underlying error class.

3. **Hook `stderr_tail` size vs. capture cap inconsistency.** §2 caps capture at 64KB per stream but includes only the "last 4KB" as `stderr_tail` in `hook_failed`; it's unclear whether 4KB is measured in bytes or after UTF-8 boundary trimming, and whether `details.truncated` reflects the 64KB cap, the 4KB tail trim, or both. Resolution: `stderr_tail` is the last 4096 bytes trimmed to the nearest valid UTF-8 boundary; `details.truncated=true` iff the 64KB capture cap was hit (independent of the 4KB tail).

4. **`compaction_locked` is exit 0 but listed in the error table.** It's documented as a warning that exits 0 yet appears alongside hard errors with a `code` field; under `--json` it's unclear whether it's emitted on stdout as `{"error": ...}` (per §1 rules) or as a normal success body, and whether tests should treat it as failure. Resolution: under `--json` emit `{"ok": true, "warning": "compaction_locked", "details": {...}}` on stdout; the `error` envelope is reserved for non-zero exits, and the error-table row is moved to a separate "warnings" subsection.

5. **MCP E2E "bounded-retry budget" duration not pinned for tests.** Test 7.5 asserts `act_next` returns `{"claimed": false, ...}` "after the bounded-retry budget" but the budget (100ms+400ms+1.6s ≈ 2.1s plus jitter) isn't restated as a test deadline, leaving implementers free to pick wildly different timeouts and flake thresholds. Resolution: test asserts total elapsed time is in `[2.0s, 3.0s]` and that exactly 3 claim attempts were made, observed via a deterministic clock injected into the server.
