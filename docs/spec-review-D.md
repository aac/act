1. **`act_next` claim-loss backoff jitter sign and distribution unspecified.** The spec says "±25% jitter" on 100ms/400ms/1.6s but doesn't define whether jitter is uniform, gaussian, or signed-symmetric, nor whether each attempt re-rolls independently. Resolution: uniform random in `[0.75x, 1.25x]` of the base delay, re-rolled per attempt.

2. **`act_block` atomicity is asserted but undefined under partial failure.** The composed tool writes `set-status` then `dep-add` and the test asserts "atomically", but there's no rule for what happens if the second op fails (rollback the first? leave it? emit which error?). Resolution: write both ops to a staging area, then `git add` + commit both in a single commit; on any failure, unstage and delete both, surface the underlying error class.

3. **Hook `stderr_tail` size vs. capture cap inconsistency.** §2 caps capture at 64KB per stream but includes only the "last 4KB" as `stderr_tail` in `hook_failed`; it's unclear whether 4KB is measured in bytes or after UTF-8 boundary trimming, and whether `details.truncated` reflects the 64KB cap, the 4KB tail trim, or both. Resolution: `stderr_tail` is the last 4096 bytes trimmed to the nearest valid UTF-8 boundary; `details.truncated=true` iff the 64KB capture cap was hit (independent of the 4KB tail).

4. **`compaction_locked` is exit 0 but listed in the error table.** It's documented as a warning that exits 0 yet appears alongside hard errors with a `code` field; under `--json` it's unclear whether it's emitted on stdout as `{"error": ...}` (per §1 rules) or as a normal success body, and whether tests should treat it as failure. Resolution: under `--json` emit `{"ok": true, "warning": "compaction_locked", "details": {...}}` on stdout; the `error` envelope is reserved for non-zero exits, and the error-table row is moved to a separate "warnings" subsection.

5. **MCP E2E "bounded-retry budget" duration not pinned for tests.** Test 7.5 asserts `act_next` returns `{"claimed": false, ...}` "after the bounded-retry budget" but the budget (100ms+400ms+1.6s ≈ 2.1s plus jitter) isn't restated as a test deadline, leaving implementers free to pick wildly different timeouts and flake thresholds. Resolution: test asserts total elapsed time is in `[2.0s, 3.0s]` and that exactly 3 claim attempts were made, observed via a deterministic clock injected into the server.
