# Atomic claim protocol

## Context
Spec §3.4 defines the exact sequence for atomic claim. The winner is the *earliest* claim op (smallest HLC tuple), not the latest — claim is a race to file first. Spec §5.B.3 clarifies that the winner is materialized as a synthetic LWW write so the rest of the LWW machinery in act-296e does not need a special case.

## Scope
- Implement `claim_protocol(issue_id, assignee, opts)` returning `{claimed bool, winner string, your_op_hash, winning_op_hash}`.
- Steps verbatim from §3.4 numbered 1–5.
- Pre-rebase HLC drift check (per §5.C.3): run the §1.5 plausibility check before `git pull --rebase`.
- Winner selection: among all claim ops visible for this issue, sorted by `(hlc.wall, hlc.logical, op_hash)`, with no intervening `close` or `reopen` between the claim and HEAD, pick the smallest tuple.
- Synthetic LWW write at fold time: when fold encounters claim ops for an issue, the apply layer (act-c9f0) consults the winner-set computation here and materializes `(assignee, status=in_progress)` from the winning claim's payload. Losing claims are recorded but their assignee/status effect is suppressed.
- `--wait` retry: backoff schedule `1s, 2s, 4s, 8s, ...`, cap at `--wait-timeout` (default 30s, max 600s). Each retry: fresh HLC via send(), fresh op_hash, full sequence (write→commit→pull→fold→winner check). Reuse existing HLC monotonicity.
- Jitter (per §5.D.1): when invoked from MCP `act_next`, multiply backoff by uniform `[0.75, 1.25]`, re-rolled per attempt. The CLI `--wait` does not jitter unless `--jitter` is passed; jitter belongs to the MCP composed tool.

## Out of scope
- MCP `act_next` composed tool itself (act-2f81).
- Auto-commit/push policy mechanics that wrap every write op (act-5ca9) — this issue only consumes that policy via `--push` flag forwarding.
- Fold checkpoint reuse during the post-rebase re-fold (act-a1f6).

## Implementation notes
- Pull-rebase rebase conflict on `.act/ops/**` is an error (`E_REBASE`); ops files are append-only by file naming so any textual conflict means corruption. Do not attempt resolution.
- The "no intervening close/reopen" predicate: walk parsed ops in HLC order; any `close` op resets the active claim window; `reopen` opens a new window. Winner is restricted to claims within the current window (or HEAD's window if open).
- Loss path: do NOT delete the losing op file. It stays in history for auditing and replay. Synthetic LWW skips its assignee/status effect.
- `--isolated` skips the pull-rebase entirely; winner selection runs on the local tree only.
- Honor universal flag rules: `--isolated --push` is exit 2; `--no-commit --push` is exit 2 (per §4 universal flags).

## Test plan
- Unit: winner selection over 5 fixtures (single claim, two-node race, race+close, race+close+reopen, race+three-node).
- Unit: backoff schedule across 6 retries returns 1s, 2s, 4s, 8s, 16s, 30s under --wait-timeout=30s.
- Integration: simulate two writers via injected HLC; first-by-tuple wins; second's run exits 5 with error code claim_lost.
- Integration: rebase conflict on .act/ops/** path → E_REBASE, exit 8.
- Property: across 100 random two-node interleavings, the winner is deterministic from the union op set (feeds act-0c76).
- Golden: claim_lost JSON envelope shape pinned in testdata/claim/lost.json.
