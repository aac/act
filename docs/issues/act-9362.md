# Op-fold algorithm core

## Context
Spec §3.2 ("Op-fold algorithm") defines the deterministic state-derivation pipeline that turns the on-disk op tree into in-memory issue state plus a derived index. This issue implements the core driver function. Per-op-type semantics live in act-c9f0; per-field LWW lives in act-296e. This issue is the scaffolding: discover, parse, sort, dispatch.

## Scope
- Implement `fold(ops_root) -> (issues, index)` as specified in §3.2 step 1..6.
- Discovery: glob `<ops_root>/*/????-??/*.json` (monthly shards under per-issue directories, per §2.5 op file naming).
- Parse each file as JSON; reject with classified errors at the gate boundary:
  - `schema_version > READER_SCHEMA_MAX` → `E_SCHEMA_TOO_NEW`
  - `writer_version > READER_WRITER_MAX` → `E_UPGRADE_REQUIRED` (maps to exit 4)
  - unknown `(op_type, op_version)` pair → `E_UNKNOWN_OP`
- Compute `op.op_hash = sha256(canonical_json(envelope-with-payload-hlc-node_id))` per §5.B.1; reuse the canonical-JSON serializer from act-b545.
- Sort by `(hlc.wall, hlc.logical, op_hash)` ascending; ties on (wall, logical) cannot occur within a node and across nodes are broken by op_hash lexicographically.
- Apply ops left-to-right via a dispatch hook `apply(state, op)`; act-c9f0 fills in the per-type bodies.
- Build `empty_issue(id)`: scalars nil, `deps={}`, `acceptance={}`, `tombstoned=false`, `last_hlc={}`.
- Surface a derived `index` struct (just an in-memory shape); SQLite materialization is act-912f.
- Refresh `last_hlc` hint from `max(op.hlc)` and expose to caller (the caller persists it to `.act/config.json`).

## Out of scope
- Per-op-type apply functions (act-c9f0).
- Per-field LWW comparison (act-296e).
- Checkpoint reuse (act-a1f6).
- SQLite index creation (act-912f).
- Atomic claim winner selection (act-9824).

## Implementation notes
- Keep fold pure: same input bytes → same output bytes. Do not depend on filesystem walk order; sort filenames before parsing for consistent error reporting only.
- The dispatch hook signature must be `apply(state IssueState, op Op) IssueState` so act-c9f0 can register per `(op_type, op_version)` handlers.
- Tombstoned issues continue to absorb subsequent ops (counted in stats) but `apply` short-circuits after the tombstone; expose a `state.tombstoned` flag from this layer so dispatch can honor it.
- Parsing errors include the offending file path in the error envelope; do not abort the whole fold on a single bad file unless the error is a hard reader-gate (E_UPGRADE_REQUIRED).

## Test plan
- Unit: empty ops_root → empty issues map, no errors.
- Unit: 3 ops for 1 issue, supplied in 6 permutations → identical sorted order, identical canonical JSON output.
- Unit: op with `writer_version = self.max + 1` → E_UPGRADE_REQUIRED.
- Unit: op with unknown `(op_type, op_version)` → E_UNKNOWN_OP.
- Property: random op streams of size 1..200 across 100 shuffles → byte-identical fold result (feeds act-a64e).
- Golden: fixed 5-op fixture under `testdata/fold/basic/` with snapshot of canonical-JSON output (feeds act-9b55).
